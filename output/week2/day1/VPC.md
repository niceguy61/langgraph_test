> **한 줄 정의:** VPC는 클라우드 환경에서 **개별 네트워크 환경을 독립적으로 구성**할 수 있는 AWS의 **가상 프라이빗 클라우드(Virtual Private Cloud)** 서비스입니다.

### 이 서비스가 해결하는 문제  
**기존의 문제점:**  
- **문제 1:** 기존에 AWS 리소스를 배포할 때는 기본적으로 공유 네트워크 환경이 적용되어, 네트워크 보안 및 트래픽 제어가 어려웠습니다. 예를 들어, 인터넷에 노출된 서버와 내부 서버가 같은 네트워크에 있어 보안 위험이 있었습니다.  
- **문제 2:** IP 주소 관리가 복잡해졌습니다. 기존에는 AWS가 자동으로 IP를 할당했지만, 특정 범위의 IP를 직접 지정해야 하는 경우가 많아 수동 관리가 필요했습니다.  
- **문제 3:** 네트워크 트래픽의 방향성 제어가 어려웠습니다. 예를 들어, 내부 서버가 인터넷에 접근해야 할 때와 외부에서 내부 서버에 접근해야 할 때의 라우팅 경로를 별도로 설정할 수 없었습니다.  

**VPC로 해결:**  
- **해결 1:** VPC는 사용자가 정의한 네트워크 환경(예: CIDR 범위)을 생성해, 외부와의 네트워크 격리를 제공합니다. 이는 네트워크 보안 및 트래픽 제어를 향상시킵니다.  
- **해결 2:** VPC에서는 사용자가 직접 CIDR 범위를 지정할 수 있어, IP 주소 관리를 유연하게 할 수 있습니다. 예를 들어, 10.0.0.0/16 범위를 지정해 특정 서브넷에 할당할 수 있습니다.  
- **해결 3:** 라우팅 테이블을 통해 트래픽 방향을 명확히 설정할 수 있습니다. 예를 들어, 인터넷 게이트웨이를 통해 외부로 트래픽을 방출하거나, NAT 게이트웨이를 통해 내부 서버가 인터넷에 접근할 수 있도록 설정할 수 있습니다.  

### 비유로 이해하기  
VPC는 **개인의 사무실 건물**을 비유할 수 있습니다. 건물 내부는 외부와 완전히 분리되어 있으며, 필요한 시점에만 출입할 수 있도록 제어됩니다. 예를 들어, 사무실 내부의 컴퓨터(서버)는 외부 인터넷에 직접 연결되지 않지만, 특정 게이트웨이를 통해 인터넷에 접근할 수 있습니다. 또한, 사무실 내부에는 여러 부서(서브넷)로 나뉘어져 각 부서의 네트워크 보안이 관리됩니다. 이처럼 VPC는 클라우드 환경에서 네트워크를 독립적으로 관리할 수 있는 '개인 공간'을 제공합니다.

## 🎯 주요 사용 시나리오 (When to Use)

| 시나리오 | 설명 | 구체적 예시 |
|---------|------|-----------|
| 시나리오 1 | 외부와의 네트워크 격리가 필요한 경우 | 금융 기관이 고객 데이터를 보호하기 위해 VPC를 사용하여 내부 서버를 외부와 분리 |
| 시나리오 2 | 여러 서브넷을 통해 애플리케이션 아키텍처 구분 | e-commerce 플랫폼에서 프론트엔드 서버와 백엔드 서버를 별도 서브넷으로 분리 |
| 시나리오 3 | 인터넷 접근이 필요하지만 외부 접근은 제한해야 하는 경우 | 내부 데이터베이스 서버가 인터넷에 노출되지 않도록 NAT 게이트웨이를 활용 |

**이럴 때 VPC를 선택하세요:**  
- ✅ **네트워크 보안이 중요한 애플리케이션**  
- ✅ **내부 서버와 외부 인터넷 간 트래픽을 분리해야 하는 시나리오**  
- ✅ **IP 주소 관리를 직접 설정해야 하는 환경**  

**이럴 때는 다른 서비스를 고려하세요:**  
- ❌ **단일 서버만 사용하고 네트워크 격리가 필요 없는 경우** → **EC2 기본 네트워크**  
- ❌ **고성능 라우팅이 필요한 경우** → **AWS Transit Gateway**  
- ❌ **다국어/다국가 네트워크 연결이 필요한 경우** → **AWS Global Accelerator**

## 🔗 연관 서비스 (Used Together With)

| 연관 서비스 | 연동 목적 | 일반적인 아키텍처 |
|------------|---------|-----------------|
| **Subnet** | VPC 내부에서 네트워크를 세분화해 보안 및 트래픽 관리 | VPC → Subnet A(퍼블릭) → EC2 인스턴스, Subnet B(프라이빗) → RDS 데이터베이스 |
| **Internet Gateway** | VPC와 인터넷 간의 연결을 제공 | VPC → Internet Gateway → 인터넷 |
| **NAT Gateway** | 내부 서버가 인터넷에 접근할 수 있도록 보안성 유지 | VPC → NAT Gateway → 인터넷 |

**자주 사용되는 아키텍처 패턴:**  
```
User → CloudFront → ELB → VPC → Subnet(퍼블릭) → EC2 인스턴스<br>
                        ↓<br>
                        Subnet(프라이빗) → RDS 데이터베이스 → NAT Gateway → 인터넷
```

## 📚 핵심 개념

### 개념 1: VPC CIDR (Classless Inter-Domain Routing)
VPC CIDR는 가상 개인 클라우드(VPC)의 IP 주소 범위를 정의하는 주요 개념입니다. CIDR 표기법은 192.168.0.0/24처럼 주소와 서브넷 마스크를 하나의 표현식으로 결합해 네트워크 범위를 정의합니다. 이는 클라우드 환경에서 네트워크를 세분화하고, 서브넷을 분할할 때 필요한 기반이 됩니다. VPC CIDR은 네트워크의 크기와 범위를 명확히 해주어, IP 주소의 낭비를 방지하고 보안 및 가용성을 높이는 데 기여합니다.

#### 왜 중요한가?
- **스케일링 유연성**: CIDR은 네트워크를 유연하게 확장하거나 축소할 수 있도록 해줍니다. 예를 들어, 10.0.0.0/16 CIDR을 사용하면 65,536개의 IP 주소를 할당할 수 있습니다.  
- **보안 및 격리**: 네트워크 범위를 명확히 설정함으로써, 내부 서브넷과 외부 인터넷 간의 트래픽을 분리해 보안을 강화할 수 있습니다.  

#### 세부 요소
| 요소 | 설명 | 예시 |
|-----|-----|-----|
| CIDR 표기법 | IP 주소와 서브넷 마스크를 결합한 표현식 | 10.0.0.0/24 |
| 범위 할당 | VPC 생성 시 정의된 IP 범위 | 192.168.1.0/24 |
| 서브넷 분할 | CIDR을 기반으로 서브넷을 나누는 방식 | 192.168.1.0/24 → 192.168.1.0/28 (16개 IP 주소) |

> **💡 Tip:** 실무에서는 CIDR 범위를 지나치게 넓게 설정하지 않도록 주의해야 합니다. 예를 들어, 10.0.0.0/8 범위는 16,777,216개의 IP 주소를 포함하므로, 과도한 IP 낭비가 발생할 수 있습니다.

---

### 개념 2: 퍼블릭/프라이빗 서브넷 구분
퍼블릭 서브넷은 인터넷에 직접 접근할 수 있는 네트워크 영역으로, 외부 사용자나 인터넷 게이트웨이와의 통신이 가능합니다. 반면, 프라이빗 서브넷은 외부 접근이 불가능하고, 내부 서비스나 데이터베이스 등의 안전한 환경을 제공합니다. 이 구분은 네트워크 보안 및 트래픽 경로 최적화에 필수적입니다.

#### 작동 원리
1. **퍼블릭 서브넷**: 인터넷 게이트웨이와 연결되어 외부 트래픽을 수신/전송합니다. 예를 들어, 웹 서버는 퍼블릭 서브넷에 배치됩니다.  
2. **프라이빗 서브넷**: NAT 게이트웨이 또는 인스턴스를 통해 외부에 트래픽을 전송할 수 있습니다. 데이터베이스나 내부 서비스는 보안을 위해 프라이빗 서브넷에 배치됩니다.  
3. **라우팅 테이블**: 퍼블릭 서브넷은 인터넷 게이트웨이를, 프라이빗 서브넷은 NAT 게이트웨이를 통해 트래픽을 전달합니다.  

> **💡 Tip:** 퍼블릭 서브넷은 외부 접근을 허용하므로, 보안 그룹과 방화벽 설정을 철저히 해야 합니다. 프라이빗 서브넷은 내부 서비스 간의 통신만 허용하는 방식으로 관리하는 것이 좋습니다.

---

### 개념 3: 라우팅 테이블 구성
라우팅 테이블은 VPC 내 트래픽이 어떻게 이동할지를 정의하는 핵심 구성 요소입니다. 이는 라우트 규칙(Routes)과 라우팅 테이블과 서브넷의 연결(Routing Table Associations)으로 구성됩니다. 라우팅 테이블은 트래픽을 목적지 서브넷, 인터넷 게이트웨이, NAT 게이트웨이 등으로 전달하는 방향성을 결정합니다.

#### 주요 특징
1. **동적 라우팅**: 트래픽이 목적지에 따라 자동으로 경로를 선택하도록 설정할 수 있습니다. 예를 들어, 인터넷 트래픽은 인터넷 게이트웨이로, 프라이빗 서브넷 트래픽은 NAT 게이트웨이로 전달됩니다.  
2. **보안 통제**: 라우팅 테이블은 트래픽의 흐름을 제어함으로써 네트워크 보안을 강화합니다. 예를 들어, 외부 트래픽을 특정 서브넷으로만 허용할 수 있습니다.  
3. **확장성**: 여러 서브넷과 게이트웨이를 연결해 네트워크를 유연하게 확장할 수 있습니다.  

> **💡 Tip:** 라우팅 테이블은 VPC 내 모든 서브넷과 연결되어 있어, 변경 시 해당 서브넷의 트래픽 흐름이 즉시 영향을 받습니다. 따라서 변경 전에 트래픽 흐름을 철저히 검토해야 합니다.

## 🖥️ AWS 콘솔에서 VPC 사용하기

### Step 1: VPC 서비스 접속  
1. **AWS Management Console 로그인**  
   - [https://console.aws.amazon.com](https://console.aws.amazon.com)에 접속 후 계정을 선택합니다.  
   - **IAM 권한 확인**: VPC 관리 권한이 있는 역할을 할당해야 합니다 (예: `AdministratorAccess`).  

2. **VPC 서비스 검색**  
   - 상단 검색창에 **"VPC"**를 입력하고, 검색 결과에서 **"Virtual Private Cloud (VPC)"**를 클릭합니다.  

> **📸 화면 확인:**  
> - **VPC 대시보드**가 열리면 정상입니다.  
> - **"My VPCs"** 탭에서 생성된 VPC 목록을 확인할 수 있습니다.  

---

### Step 2: [주요 작업 1 - 리소스 생성]  
1. **VPC 생성**  
   - **"Create VPC"** 버튼 클릭 → **CIDR 설정** (예: `10.0.0.0/16`)  
   - **VPC 이름** 입력 (예: `My-VPC`)  
   - **DNS 호스트 이름**을 `enable`로 설정 (DNS 서버 자동 설정)  
   - **"Create"** 버튼 클릭  

2. **퍼블릭 서브넷 생성**  
   - **"Subnets"** 탭 → **"Create subnet"**  
   - **VPC 선택** → **CIDR 입력** (예: `10.0.1.0/24`)  
   - **Subnet 이름** 입력 (예: `Public-Subnet`)  
   - **"Create"** 버튼 클릭  

3. **프라이빗 서브넷 생성**  
   - **"Create subnet"** → **CIDR 입력** (예: `10.0.2.0/24`)  
   - **Subnet 이름** 입력 (예: `Private-Subnet`)  
   - **"Create"** 버튼 클릭  

> **📸 화면 확인:**  
> - **Subnets 탭**에서 생성된 서브넷이 표시되고, **VPC와 연결**되어 있는지 확인합니다.  

---

### Step 3: [주요 작업 2 - 설정/구성]  
1. **인터넷 게이트웨이 연결**  
   - **"Internet Gateways"** 탭 → **"Create internet gateway"**  
   - **IGW 이름** 입력 (예: `My-IGW`)  
   - **"Create"** 버튼 클릭 → 생성된 IGW를 **VPC에 연결**합니다.  

2. **라우팅 테이블 구성**  
   - **"Route tables"** 탭 → **"Create route table"**  
   - **VPC 선택** → **"Routes"** 섹션에서 **"Route"** 추가  
     - **Destination CIDR**: `0.0.0.0/0`  
     - **Target**: **Internet Gateway** 선택  
   - **"Save"** 버튼 클릭  

3. **NAT 게이트웨이 설정**  
   - **"NAT Gateways"** 탭 → **"Create NAT gateway"**  
   - **Subnet 선택** (프라이빗 서브넷)  
   - **Elastic IP 할당** (AWS가 자동 할당)  
   - **"Create"** 버튼 클릭  

> **⚠️ 주의:**  
> - **NAT 게이트웨이**는 프라이빗 서브넷에서만 생성 가능합니다.  
> - **라우팅 테이블**에서 **Private Subnet**에 대한 라우트를 **NAT Gateway**로 설정해야 합니다.  

---

### Step 4: 설정 확인 및 테스트  
1. **리소스 확인 방법**  
   - **VPC 대시보드**에서 생성된 VPC, 서브넷, IGW, NAT Gateway를 확인합니다.  
   - **"Resources"** 탭에서 각 리소스의 **Status**가 `available`인지 확인합니다.  

2. **정상 동작 테스트**  
   - **EC2 인스턴스 생성** (프라이빗 서브넷에) → SSH 연결 시 **NAT Gateway**를 통해 외부 접근 가능해야 합니다.  
   - **CLI 테스트**:  
     ```bash
     aws ec2 describe-vpcs --vpc-ids vpc-0abcdef1234567890
     aws ec2 describe-subnets --subnet-ids subnet-0123456789abcdef0
     ```  

> **💡 Tip:**  
> - **AWS Free Tier**는 1개 VPC, 2개 서브넷, 1개 NAT Gateway까지 무료입니다.  
> - **비용 최소화**: NAT Gateway 대신 NAT 인스턴스를 사용할 수 있으나, 관리 부담이 더 큽니다.

### 사전 준비
```bash
# AWS CLI 버전 확인
aws --version

# AWS 자격 증명 확인
aws sts get-caller-identity

```

**설명:**  
AWS CLI를 사용하기 전에 버전을 확인하고, 자격 증명이 정상적으로 설정되어 있는지, 현재 리전이 올바른지 확인해야 합니다. `aws configure get region` 명령어로 현재 설정된 리전을 확인하고, 필요 시 `aws configure set region us-east-1`처럼 리전을 변경할 수 있습니다.

---

### 예제 1: VPC 리소스 조회
```bash
# VPC 목록 조회
aws ec2 describe-vpcs --query 'Vpcs[*].{ID:VpcId, Name:Tags[?Key==`Name`].Value|[0], CIDR: CidrBlock}' --output table
```

**옵션 설명:**
| 옵션 | 설명 | 예시 값 |
|-----|------|--------|
| `--query` | 필터링 및 형식 조정 | `'Vpcs[*].{ID:VpcId, Name:Tags[?Key==`Name`].Value|[0], CIDR: CidrBlock}'` |
| `--output` | 출력 형식 | `table` (테이블 형식), `json` (JSON 형식) |

**예상 출력:**
```
|    ID      |      Name      |        CIDR        |
|------------|----------------|---------------------|
| vpc-123456 | MyVPC-Example  | 10.0.0.0/16        |
```

**설명:**  
`describe-vpcs` 명령어는 VPC 목록을 조회합니다. `--query` 파라미터를 사용해 필터링하고, `--output table`로 테이블 형식으로 출력합니다. `Tags[?Key==`Name`]`는 `Name` 태그 값을 추출합니다.

---

### 예제 2: VPC 리소스 생성
```bash
# VPC 생성
aws ec2 create-vpc \
    --cidr 10.0.0.0/16 \
    --tags Key=Name,Value=MyVPC-Example
```

**필수 옵션:**
- `--cidr`: VPC의 IP 범위 (예: `10.0.0.0/16`)
- `--tags`: 리소스에 태그 추가 (이름 지정)

**예상 출력:**
```json
{
    "Vpc": {
        "VpcId": "vpc-1234567890abcdef0",
        "CidrBlock": "10.0.0.0/16",
        "State": "pending"
    }
}
```

**설명:**  
`create-vpc` 명령어는 지정된 CIDR 범위로 VPC를 생성합니다. `--tags` 옵션으로 리소스 이름을 지정하면, 관리자가 쉽게 리소스를 식별할 수 있습니다. 생성 후 `describe-vpcs`로 확인할 수 있습니다.

---

### 예제 3: VPC 리소스 수정
```bash
# VPC 태그 수정
aws ec2 create-tags \
    --resources vpc-1234567890abcdef0 \
    --tags Key=Name,Value=UpdatedVPC-Example
```

**설명:**  
VPC의 태그는 `create-tags` 명령어로 수정할 수 있습니다. `--resources`에 수정할 VPC ID를, `--tags`에 새로운 태그를 지정합니다. VPC 자체의 IP 범위나 네트워크 설정은 수정할 수 없으며, 이는 VPC 생성 시 결정해야 합니다.

---

### 예제 4: VPC 리소스 삭제
```bash
# VPC 삭제
aws ec2 delete-vpc --vpc-id vpc-1234567890abcdef0

# 삭제 확인
aws ec2 describe-vpcs --vpc-ids vpc-1234567890abcdef0
```

> **⚠️ 주의:** VPC를 삭제하면 해당 VPC에 연결된 모든 리소스(서브넷, 인스턴스 등)도 함께 삭제됩니다. 삭제 전에 리소스 상태를 점검하고, `--dry-run` 옵션으로 시뮬레이션할 수 있습니다.

**삭제 시 주의사항:**  
- VPC에 연결된 인터넷 게이트웨이, NAT 게이트웨이, 서브넷이 있는 경우 삭제 실패
- 삭제 후에는 `describe-vpcs`로 확인해 삭제 상태를 확인해야 합니다.

---

### 자주 사용하는 명령어 정리
```bash
# 조회
aws ec2 describe-vpcs --query 'Vpcs[*].{ID:VpcId, Name:Tags[?Key==`Name`].Value|[0], CIDR: CidrBlock}' --output table
aws ec2 describe-vpcs --vpc-ids vpc-1234567890abcdef0

# 생성
aws ec2 create-vpc --cidr 10.0.0.0/16 --tags Key=Name,Value=MyVPC

# 수정
aws ec2 create-tags --resources vpc-1234567890abcdef0 --tags Key=Name,Value=UpdatedVPC

# 삭제
aws ec2 delete-vpc --vpc-id vpc-1234567890abcdef0
```

**팁:**  
- VPC 생성 시 `--cidr`를 정확히 설정해 네트워크 분할을 방지합니다.
- `--dry-run` 옵션으로 삭제 시뮬레이션: `aws ec2 delete-vpc --vpc-id vpc-1234567890abcdef0 --dry-run`

## 🎯 SAA-C03 시험 핵심 포인트

### 시험에서 자주 출제되는 VPC 포인트

#### 📌 핵심 출제 포인트 TOP 5
1. **VPC CIDR 설정 및 IP 주소 충돌 방지**  
   - 설명: VPC의 CIDR 범위는 네트워크 설계의 기초로, IP 주소 충돌을 방지하고 자원 최적화에 필수적입니다. 시험에서 VPC의 CIDR 범위가 다른 VPC와 겹치는 경우, **CIDR 겹치기** 문제로 출제됩니다.  
   - 키워드: `CIDR`, `IP 주소 충돌`, `VPC 설계`

2. **퍼블릭/프라이빗 서브넷 구분 및 보안 정책**  
   - 설명: 퍼블릭 서브넷은 인터넷 접근이 필요하고, 프라이빗 서브넷은 내부 네트워크에만 연결됩니다. 보안 정책에서 **서브넷 유형 선택**이 시험에서 주요 포인트로 출제됩니다.  
   - 키워드: `서브넷 유형`, `보안 정책`, `접근 제어`

3. **라우팅 테이블 구성 및 라우트 전파**  
   - 설명: 라우팅 테이블은 VPC 내외부 트래픽 경로를 정의합니다. **라우트 전파 방식** 및 라우팅 테이블 구성 오류는 시험에서 빈도가 높은 주제입니다.  
   - 키워드: `라우팅 테이블`, `라우트 전파`, `트래픽 경로`

4. **인터넷 게이트웨이의 역할 및 NAT 게이트웨이 vs NAT 인스턴스**  
   - 설명: 인터넷 게이트웨이는 VPC와 인터넷 연결을 제공하고, NAT 게이트웨이는 프라이빗 서브넷에서 인터넷 접근을 지원합니다. **NAT 게이트웨이 vs NAT 인스턴스**의 성능/비용 차이는 시험에서 자주 비교됩니다.  
   - 키워드: `NAT`, `인터넷 게이트웨이`, `비용 효율성`

5. **VPC의 제한사항 및 확장성 제어**  
   - 설명: VPC는 최대 5개의 네트워크 ACL, 200개의 서브넷, 100개의 라우팅 테이블 등 제한사항이 존재합니다. **VPC 한계 이해**는 시험에서 필수적으로 출제됩니다.  
   - 키워드: `VPC 한계`, `확장성`, `네트워크 ACL`

#### ⚠️ 시험에서 자주 나오는 함정
| 함정 유형 | 설명 | 올바른 답변 |
|----------|------|------------|
| 함정 1 | "NAT 인스턴스는 NAT 게이트웨이보다 비용이 더 비싸다"라는 문장에서 헷갈릴 수 있음 | **NAT 인스턴스는 NAT 게이트웨이보다 비용이 더 비싸다** |
| 함정 2 | VPC CIDR이 다른 VPC와 겹치는 경우, "CIDR 충돌"이 발생한다고 오해할 수 있음 | **CIDR 겹치기로 인해 VPC 연결 불가** |
| 함정 3 | 라우팅 테이블에서 "0.0.0.0/0" 라우트를 추가하지 않으면 인터넷 접근이 불가능하다는 사실을 모르는 경우 | **0.0.0.0/0 라우트 추가 필수** |

#### 🔄 VPC vs 비슷한 서비스 비교 (시험 단골!)
| 비교 항목 | VPC | 대안 서비스 | 선택 기준 |
|----------|----------|-----------|----------|
| 용도 | VPC는 가상 네트워크 인프라 구축 | Direct Connect는 데이터센터와 AWS 간 고속 연결 | **지역 내부 네트워크 설계** vs **외부 네트워크 연결** |
| 확장성 | 최대 200개 서브넷, 100개 라우팅 테이블 | VPC Peering은 VPC 간 연결 | **내부 네트워크 확장** vs **다중 VPC 통신** |
| 비용 | 서브넷/게이트웨이 수에 따라 변동 | NAT 게이트웨이는 고정 비용, NAT 인스턴스는 CPU/메모리 비용 | **비용 최적화** vs **성능/가용성** |
| 지연시간 | 라우팅 테이블 설정에 따라 차이 | VPC Peering은 라우팅 테이블 추가로 지연 최소화 | **트래픽 지연 최소화** vs **네트워크 복잡성** |
