# Subnet 완전 정복

## 📌 핵심 목적 (What & Why)

> **한 줄 정의:** Subnet은 AWS의 **VPC 내 IP 주소 범위를 관리하고 네트워크 보안을 강화**하기 위한 서비스입니다.

### 이 서비스가 해결하는 문제
**기존의 문제점:**
- **문제 1:** 기존 네트워크 설계는 IP 주소 할당이 수동적이었고, IP 주소 누수나 중복이 자주 발생했습니다. 예를 들어, 여러 서버가 같은 IP 범위를 사용해 네트워크 충돌이 발생했습니다.
- **문제 2:** 서버의 보안 관리가 복잡했습니다. 모든 서버가 동일한 네트워크에 있어 외부 공격에 취약했으며, 내부 네트워크에서의 트래픽 통제가 어려웠습니다.
- **문제 3:** 가상 네트워크 확장성이 부족했습니다. 기존 네트워크는 서버 수 증가 시 IP 주소 할당이 어려워졌고, 성능 저하가 발생했습니다.

**Subnet으로 해결:**
- **해결 1:** Subnet은 VPC 내 IP 주소 범위를 세분화해 동적 IP 할당을 가능하게 합니다. 예를 들어, 퍼블릭 서브넷은 외부 인터넷 접근이 가능하고, 프라이빗 서브넷은 내부 통신만 허용해 보안을 강화합니다.
- **해결 2:** 서브넷 간 네트워크 보안 정책을 구분해 트래픽 통제가 가능합니다. 예를 들어, 특정 서브넷에만 특정 포트를 오픈해 보안 위험을 줄일 수 있습니다.
- **해결 3:** 서브넷을 추가/삭제해 유연한 네트워크 확장을 가능하게 합니다. 서버 수가 증가해도 IP 주소 할당이 자동화되어 네트워크 성능을 유지할 수 있습니다.

### 비유로 이해하기
**아파트 단지의 방**을 생각해보세요. 각 아파트(서브넷)는 독립적인 방으로, 외부(인터넷)에 연결되는 출입구(인터넷 게이트웨이)와 내부(프라이빗 네트워크)의 연결을 구분합니다. 아파트 간은 단지 내 통로(라우팅 테이블)로 연결되며, 각 방(서버)은 보안 문(보안 그룹)으로 보호됩니다. 이처럼 Subnet은 네트워크를 구조화해 보안과 효율성을 높입니다.

## 🎯 주요 사용 시나리오 (When to Use)

| 시나리오 | 설명 | 구체적 예시 |
|---------|------|-----------|
| 시나리오 1 | **외부 인터넷 접근이 필요한 서버와 내부 시스템을 분리** | 클라우드 기반 웹 애플리케이션(퍼블릭 서브넷)과 데이터베이스(프라이빗 서브넷) 분리 |
| 시나리오 2 | **내부 시스템 간 네트워크 보안 정책 적용** | 기업 내부 서버(프라이빗 서브넷) 간 통신 제한 및 외부 접근 차단 |
| 시나리오 3 | **하이브리드 클라우드 환경 구축** | 데이터센터(온프레미스)와 AWS 서브넷 간 네트워크 연결 |

**이럴 때 Subnet을 선택하세요:**
- ✅ **네트워크 보안이 중요한 시스템 구축 시**  
- ✅ **서버 수가 급증할 수 있는 유연한 환경에서**  
- ✅ **외부 인터넷 접근과 내부 통신을 구분해야 할 때**

**이럴 때는 다른 서비스를 고려하세요:**
- ❌ **전체 네트워크 설계가 필요한 경우 → VPC**  
- ❌ **DNS 관리가 필요한 경우 → Route 53**  
- ❌ **서버 보안 정책이 복잡한 경우 → Security Groups**

## 🔗 연관 서비스 (Used Together With)

| 연관 서비스 | 연동 목적 | 일반적인 아키텍처 |
|------------|---------|-----------------|
| **VPC** | Subnet은 VPC 내에서 IP 주소 범위를 관리 | `User → Internet Gateway → Public Subnet → EC2` |

**자주 사용되는 아키텍처 패턴:**
```
User → CloudFront → Public Subnet (NAT Gateway) → Private Subnet (EC2) → RDS (Private Subnet)
```

## 📚 핵심 개념

### 개념 1: Subnet의 정의 및 역할  
Subnet은 AWS Virtual Private Cloud(VPC) 내에서 정의된 IP 주소 범위를 기반으로 구성된 논리적 네트워크 영역입니다. VPC는 대규모 IP 범위를 가지며, 이를 여러 Subnet으로 세분화하여 네트워크 보안, 트래픽 관리, 자원 분리 등을 용이하게 합니다. Subnet은 CIDR 블록(예: 10.0.1.0/24)을 통해 IP 주소 범위를 정의하고, 각 Subnet은 라우팅 테이블과 연동되어 트래픽을 제어합니다.  
Subnet은 네트워크 보안 정책을 적용하고, 특정 서브넷에만 제한된 액세스를 제공할 수 있어 기업의 데이터 보호에 필수적입니다. 또한, 서브넷은 VPC 내에서 자원(예: EC2 인스턴스, RDS)을 효율적으로 배치하고, 네트워크 성능 최적화에 기여합니다.  

#### 왜 중요한가?  
- **네트워크 보안 강화**: 특정 서브넷에만 액세스를 제한해 내부 네트워크를 보호합니다.  
- **트래픽 제어 및 최적화**: 라우팅 테이블을 통해 특정 서브넷의 트래픽 경로를 정의해 네트워크 효율성을 높입니다.  
- **리소스 분리**: 개발, 테스트, 프로덕션 환경을 별도의 서브넷으로 구분해 운영 관리 편의성을 제공합니다.  

#### 세부 요소  
| 요소 | 설명 | 예시 |
|-----|-----|-----|
| **CIDR 블록** | IP 주소 범위를 정의하는 기술 (예: 10.0.1.0/24) | `10.0.1.0/24`로 254개의 IP 주소를 할당 |
| **라우팅 테이블** | 서브넷의 트래픽을 어디로 전달할지 정의 | `rtb-12345678`이라는 라우팅 테이블과 연결 |
| **사용 가능한 존(Zone)** | 가용성 존에 따라 서브넷을 배치해 고가용성 보장 | `us-east-1a`, `us-east-1b` 등에 분산 배치 |

> **💡 Tip:** 서브넷은 VPC의 IP 범위를 세분화해 네트워크 보안과 효율성을 극대화하는 핵심 요소로, 보안 그룹과 결합해 더욱 강력한 보호를 제공합니다.

---

### 개념 2: 퍼블릭/프라이빗 서브넷의 구분 및 작동 원리  
퍼블릭 서브넷은 인터넷에 직접 연결되어 외부와의 통신이 가능한 네트워크 영역이고, 프라이빗 서브넷은 인터넷에 직접 연결되지 않으며 내부 네트워크만을 위한 영역입니다. 퍼블릭 서브넷은 인터넷 게이트웨이를 통해 외부 트래픽을 수신하고, 프라이빗 서브넷은 NAT 게이트웨이 또는 NAT 인스턴스를 통해 인터넷 접근을 제어합니다.  

#### 작동 원리  
1. **퍼블릭 서브넷**: EC2 인스턴스가 인터넷에 직접 접근할 수 있도록 인터넷 게이트웨이와 연결됩니다.  
2. **프라이빗 서브넷**: EC2 인스턴스는 NAT 게이트웨이를 통해 인터넷에 접근 가능하지만, 외부에서 해당 인스턴스에 직접 접근할 수 없습니다.  
3. **트래픽 경로**: 퍼블릭 서브넷의 트래픽은 인터넷 게이트웨이를 통해 외부로 전송되고, 프라이빗 서브넷의 트래픽은 NAT 게이트웨이를 통해 외부로 변환됩니다.  

> **💡 Tip:** 웹 서버는 퍼블릭 서브넷에 배치하고, 데이터베이스는 프라이빗 서브넷에 배치해 보안을 강화하는 것이 일반적입니다.

---

### 개념 3: Subnet 생성 및 관리의 주요 특징  
Subnet은 VPC 내에서 IP 주소를 관리하고, 네트워크 보안 및 트래픽 제어를 위한 필수 구성 요소입니다. Subnet을 생성할 때는 CIDR 블록을 설정하고, 라우팅 테이블과 연동해 트래픽 경로를 정의해야 합니다. 또한, 서브넷은 가용성 존을 기준으로 분산 배치해 고가용성 환경을 구축할 수 있습니다.  

#### 주요 특징  
1. **CIDR 블록 관리**: 서브넷의 IP 주소 범위를 정의해 네트워크를 세분화합니다. 예를 들어, `10.0.1.0/24`로 254개의 IP 주소를 할당할 수 있습니다.  
2. **라우팅 테이블 연동**: 각 서브넷은 라우팅 테이블을 통해 트래픽을 제어해 네트워크 효율성을 높입니다.  
3. **가용성 존 분산**: 서브넷을 여러 가용성 존에 배치해 장애 시 다른 존에서 계속 운영할 수 있습니다.  

> **💡 Tip:** 서브넷을 생성할 때는 프리티어로 750시간까지 무료로 사용할 수 있으므로, 실습 시 다양한 CIDR 블록과 라우팅 테이블을 테스트해 보세요.

## 🖥️ AWS 콘솔에서 Subnet 사용하기

### Step 1: Subnet 서비스 접속  
1. AWS Management Console에 로그인합니다.  
   - URL: https://console.aws.amazon.com  
   - AWS 계정이 활성화된 경우, 자동으로 리전 선택 화면이 나타납니다.  
2. 상단 검색창에 **"Subnet"**을 입력하고, 검색 결과에서 **"Subnets"**를 클릭합니다.  
3. "Subnets" 메뉴는 **VPC에 속한 서브넷 목록**을 표시합니다.  

> **📸 화면 확인:** "Subnets" 대시보드가 열리고, 생성된 서브넷 목록이 표시되면 정상입니다.  

---

### Step 2: [주요 작업 1 - 리소스 생성]  
1. **서브넷 생성하기**  
   - 왼쪽 메뉴에서 **"Subnets" → "Create subnet"**을 클릭합니다.  
   - **VPC 선택**: 생성할 서브넷에 할당할 VPC를 선택합니다.  
   - **서브넷 이름**: 유의미한 이름을 입력합니다 (예: `Public-Subnet-1`).  
   - **CIDR 범위**:  
     - **퍼블릭 서브넷**: `10.0.1.0/24` 등 공개 인터넷에 접근 가능한 CIDR를 입력합니다.  
     - **프라이빗 서브넷**: `10.0.2.0/24` 등 내부 네트워크만 사용하는 CIDR를 입력합니다.  
   - **사용할 가용성 존**: 서버 가용성 존을 선택합니다 (예: `us-east-1a`).  

2. **서브넷 유형 설정**  
   - **Public Subnet**: 인터넷 게이트웨이와 라우팅 테이블 연결이 자동 생성됩니다.  
   - **Private Subnet**: NAT 게이트웨이 또는 인스턴스를 통해 외부 접근이 가능합니다.  

3. **확인 및 생성**  
   - **Next** 버튼을 클릭하여 설정을 확인하고, **Create**를 누릅니다.  
   - 생성 완료 후, 서브넷 목록에 새 서브넷이 표시됩니다.  

> **📸 화면 확인:** "Create subnet" 화면에서 CIDR 범위와 VPC 선택이 제대로 입력되었는지 확인합니다.  

---

### Step 3: [주요 작업 2 - 설정/구성]  
1. **라우팅 테이블 연결**  
   - 왼쪽 메뉴에서 **"Route tables" → "Edit routes"**를 클릭합니다.  
   - **Public Subnet**에 연결된 라우팅 테이블에서 **"Route"**를 추가합니다.  
   - **Destination**: `0.0.0.0/0` (모든 트래픽)  
   - **Target**: 생성한 인터넷 게이트웨이를 선택합니다.  

2. **NAT 게이트웨이 설정 (프라이빗 서브넷용)**  
   - **NAT Gateway** 생성 후, 프라이빗 서브넷에 연결된 라우팅 테이블에서 **"Route"**를 추가합니다.  
   - **Destination**: `0.0.0.0/0`  
   - **Target**: 생성한 NAT 게이트웨이를 선택합니다.  

3. **서브넷 상태 확인**  
   - 서브넷 목록에서 **"State"** 열을 확인합니다.  
   - **"available"** 상태일 경우 정상적으로 작동합니다.  

> **⚠️ 주의:** CIDR 범위가 중복되지 않도록 주의하세요. 예: `10.0.1.0/24`는 다른 서브넷과 중복되면 네트워크 갈등이 발생할 수 있습니다.  

---

### Step 4: 설정 확인 및 테스트  
1. **리소스 확인 방법**  
   - **Subnets 대시보드**: 생성된 서브넷이 목록에 포함되어 있는지 확인합니다.  
   - **Route tables 대시보드**: 라우팅 테이블에 인터넷 게이트웨이 또는 NAT 게이트웨이가 연결되었는지 확인합니다.  

2. **상태 확인 방법**  
   - 서브넷 상태가 **"available"**인지, **"pending"**인지 확인합니다.  
   - 상태가 **"available"**이면 정상적으로 작동합니다.  

3. **정상 동작 테스트**  
   - **CLI 명령어**:  
     ```bash
     aws ec2 describe-subnets --subnet-ids <subnet-id>
     ```  
     - 생성된 서브넷의 **CIDR**과 **VPC** 정보를 확인합니다.  
   - **SSH 테스트**:  
     - 퍼블릭 서브넷에서 인스턴스를 생성하고, 프라이빗 서브넷 인스턴스에 SSH 연결을 시도합니다.  
     - NAT 게이트웨이가 제대로 작동하면 연결이 성공해야 합니다.  

---

### 📌 실무 팁  
- **프리티어 활용**: VPC 및 서브넷은 AWS 프리티어에서 **무료로 사용**할 수 있습니다.  
- **비용 관리**: 서브넷의 **CIDR 범위를 너무 넓게 설정하지 마세요**. IP 주소를 낭비하지 않도록 주의하세요.  
- **자동화**: CLI 또는 Terraform으로 서브넷 생성을 자동화할 수 있습니다.  

> **💡 Tip:** AWS 콘솔에서 "Subnets"를 생성할 때, **사용자 정의 네트워크 설정**을 사용하면 보안 및 관리 효율성이 높아집니다.

## ⌨️ AWS CLI로 Subnet 사용하기

### 사전 준비
```bash
# AWS CLI 버전 확인
aws --version

# AWS 자격 증명 확인
aws sts get-caller-identity

# 현재 리전 확인
aws configure get region
```

> **💡 Tip:** CLI 버전이 1.28.0 이상이고, AWS 자격 증명이 정상적으로 설정되어야 Subnet 작업이 가능합니다. 리전은 `us-east-1` 또는 `ap-northeast-1` 등에서 실행 가능합니다.

---

### 예제 1: Subnet 리소스 조회
```bash
# Subnet 목록 조회 (필터링 가능)
aws ec2 describe-subnets --query 'Subnets[*].{ID:SubnetId, Name:Tags[?Key==`Name`].Value[0], CIDR:CidrBlock}' --output table
```

**옵션 설명:**
| 옵션 | 설명 | 예시 값 |
|-----|------|--------|
| `--query` | 결과 필터링 (예: Subnet ID, 이름, CIDR) | `'Subnets[*].{ID:SubnetId, Name:Tags[?Key==`Name`].Value[0], CIDR:CidrBlock}'` |
| `--output` | 출력 형식 (table, json, text) | `table` |

**예상 출력:**
```
|    ID      |     Name     |         CIDR         |
|------------|--------------|----------------------|
| subnet-123 | Public-Subnet | 10.0.1.0/24         |
| subnet-456 | Private-Subnet| 10.0.2.0/24         |
```

> **💡 Tip:** `Tags[?Key==`Name`]`으로 Subnet 이름을 필터링할 수 있습니다. `Name` 태그가 없는 경우 `Name` 열은 비어 있습니다.

---

### 예제 2: Subnet 리소스 생성
```bash
# Subnet 생성 (VPC ID와 CIDR 필수)
aws ec2 create-subnet \
    --vpc-id vpc-0abcdef1234567890 \
    --cidr-block 10.0.1.0/24 \
    --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=Public-Subnet}]"
```

**필수 옵션:**
- `--vpc-id`: 생성할 VPC ID (예: `vpc-0abcdef1234567890`)
- `--cidr-block`: Subnet CIDR (예: `10.0.1.0/24`)
- `--tag-specifications`: Subnet 이름 설정 (명시적 태그 추가)

**예상 출력:**
```json
{
    "Subnet": {
        "SubnetId": "subnet-1234567890abcdef0",
        "VpcId": "vpc-0abcdef1234567890",
        "CidrBlock": "10.0.1.0/24",
        "State": "pending"
    }
}
```

> **⚠️ 주의:** VPC가 먼저 생성되어야 Subnet 생성이 가능합니다. CIDR은 VPC CIDR 범위 내에 있어야 합니다.

---

### 예제 3: Subnet 리소스 수정
```bash
# Subnet 상태 수정 (예: 상태 확인)
aws ec2 describe-subnets --subnet-ids subnet-1234567890abcdef0
```

> **💡 Tip:** Subnet의 기본 속성(예: CIDR, VPC ID)은 수정 불가능합니다. 상태(`State`)만 확인하거나, `modify-subnet-attribute` 명령어로 특정 속성 변경이 가능합니다.

---

### 예제 4: Subnet 리소스 삭제
```bash
# Subnet 삭제
aws ec2 delete-subnet --subnet-id subnet-1234567890abcdef0

# 삭제 확인
aws ec2 describe-subnets --subnet-ids subnet-1234567890abcdef0
```

> **⚠️ 주의:** 삭제는 되돌릴 수 없습니다. Subnet에 연결된 인스턴스나 라우팅 테이블이 있는 경우 삭제 실패합니다.

---

### 자주 사용하는 명령어 정리
```bash
# 조회
aws ec2 describe-subnets --subnet-ids subnet-1234567890abcdef0
aws ec2 describe-subnets --filters "Name=vpc-id,Values=vpc-0abcdef1234567890"

# 생성
aws ec2 create-subnet --vpc-id vpc-0abcdef1234567890 --cidr-block 10.0.1.0/24

# 삭제
aws ec2 delete-subnet --subnet-id subnet-1234567890abcdef0
```

> **💡 Tip:** Subnet 생성 시 `--tag-specifications`으로 이름을 지정해 관리성을 높이세요. AWS Free Tier는 Subnet 생성이 포함됩니다 (VPC 제한 내).

## 🎯 SAA-C03 시험 핵심 포인트

### 시험에서 자주 출제되는 Subnet 포인트

#### 📌 핵심 출제 포인트 TOP 5
1. **포인트 1: 퍼블릭/프라이빗 서브넷의 구분 및 보안 역할**  
   - 설명: 시험에서 퍼블릭 서브넷은 인터넷 접근이 가능해야 하며, 프라이빗 서브넷은 내부 네트워크만 사용하는 점이 주요 포인트입니다. 보안 정책에 따라 퍼블릭 서브넷은 보안 그룹과 ACL을 통해 제어해야 하며, 프라이빗 서브넷은 VPC 내부 통신에만 사용됩니다.  
   - 키워드: `보안 정책`, `ACL`, `보안 그룹`

2. **포인트 2: CIDR 블록 및 서브넷 크기 계산**  
   - 설명: CIDR 블록을 분할해 서브넷을 구성할 때, 16, 24, 12 등 다양한 전달 길이를 사용하는 경우가 많습니다. 시험에서는 서브넷 크기(예: /24는 256개 IP 주소)와 CIDR 겹치지 않는지 확인하는 문제가 자주 출제됩니다.  
   - 키워드: `CIDR 계산`, `서브넷 크기`, `IP 주소 범위`

3. **포인트 3: 라우팅 테이블 구성 및 트래픽 흐름**  
   - 설명: 서브넷에 할당된 IP 주소 범위와 라우팅 테이블의 라우트 규칙이 트래픽 흐름을 결정합니다. 퍼블릭 서브넷은 인터넷 게이트웨이, 프라이빗 서브넷은 NAT 게이트웨이 또는 라우터를 통해 인터넷 접근을 설정하는 점이 중요합니다.  
   - 키워드: `라우팅 테이블`, `트래픽 흐름`, `NAT 게이트웨이`

4. **포인트 4: NAT 게이트웨이 vs NAT 인스턴스의 차이**  
   - 설명: NAT 게이트웨이는 고가용성과 보안성이 우수하지만, NAT 인스턴스는 비용이 낮고 유연한 설정이 가능합니다. 시험에서는 서브넷에서 인터넷 액세스를 제공하는 방식에 따라 선택하는 문제를 자주 출제합니다.  
   - 키워드: `NAT 게이트웨이`, `NAT 인스턴스`, `고가용성`

5. **포인트 5: 서브넷 제한사항 및 CIDR 겹치기 문제**  
   - 설명: 한 VPC당 최대 200개 서브넷 제한, CIDR 범위 중복 시 네트워크 간섭 발생 등 제한사항은 시험에서 자주 다룹니다. 예를 들어, 서브넷 CIDR이 중복되면 VPC 내부 통신이 실패할 수 있습니다.  
   - 키워드: `서브넷 제한`, `CIDR 중복`, `네트워크 간섭`

#### ⚠️ 시험에서 자주 나오는 함정
| 함정 유형 | 설명 | 올바른 답변 |
|----------|------|------------|
| 함정 1 | "퍼블릭 서브넷은 항상 인터넷 게이트웨이를 필요로 한다"는 오해 | 라우팅 테이블의 라우트 규칙이 인터넷 접근 여부를 결정합니다. |
| 함정 2 | NAT 인스턴스는 NAT 게이트웨이보다 비용이 낮다 | NAT 인스턴스는 EC2 인스턴스로, 장기적으로 비용이 더 높습니다. |
| 함정 3 | "서브넷 CIDR 중복은 VPC 생성 시 자동 검출된다"는 오해 | 수동으로 확인해야 하며, 중복 시 VPC 생성이 실패합니다. |

#### 🔄 Subnet vs 비슷한 서비스 비교 (시험 단골!)
| 비교 항목 | Subnet | 대안 서비스 | 선택 기준 |
|----------|----------|-----------|----------|
