# NACL 완전 정복

## 📌 핵심 목적 (What & Why)

> **한 줄 정의:** NACL는 **네트워크 트래픽 통제**를 위한 AWS의 **네트워크 보안** 서비스입니다.

### 이 서비스가 해결하는 문제
**기존의 문제점:**
- **문제 1:** 전통적인 보안 그룹은 **상태 기반**(stateful) 규칙으로, 허용된 트래픽이 들어오면 자동으로 응답 트래픽을 허용하는 문제가 있었습니다. 이로 인해 **미국에서의 해킹 시도** 같은 상황에서 악성 트래픽이 무기력하게 허용될 수 있었습니다.
- **문제 2:** **IP 주소 변경** 또는 **서브넷 확장** 시 기존 보안 규칙을 매번 수정해야 하며, **관리 복잡도**가 높았습니다. 예를 들어, 새로운 서버를 추가하면 모든 규칙을 점검해야 했습니다.
- **문제 3:** **트래픽 로그 분석**이 별도의 서비스를 필요로 하여, **보안 모니터링**이 복잡하고 비용이 많이 들었습니다.

**NACL로 해결:**
- **해결 1:** **상태 무기**(stateless) 방식으로, 모든 트래픽이 **명시적으로 허용/거부**되어야 하므로, **해킹 시도**에도 정확한 보안을 제공합니다. 예를 들어, SSH 로그인 시도가 실패해도 응답 트래픽을 무조건 허용하지 않습니다.
- **해결 2:** **서브넷별로 별도 규칙**을 정의할 수 있어, 서버 추가 시 **규칙 수정이 필요하지 않아** 관리 효율성이 높아집니다. 예를 들어, 새로운 서버가 추가되면 해당 서브넷의 NACL만 수정하면 됩니다.
- **해결 3:** **VPC Flow Logs**와 연동하여 **트래픽 로그를 실시간으로 분석**할 수 있어, 보안 모니터링이 단순화됩니다. 예를 들어, 특정 IP의 이상 트래픽이 감지되면 즉시 대응이 가능합니다.

### 비유로 이해하기
**NACL는 건물의 문 경비원처럼 작동합니다.** 사람의 출입을 기록하고, 특정 사람만 허용하고, 불법 접근 시 경고를 내립니다. 이와 마찬가지로, NACL은 **네트워크 트래픽의 출입을 통제**하며, **불법적인 데이터 전송을 차단**합니다. 이는 **보안 관리의 첫 번째 줄 방어선**으로, 모든 트래픽이 NACL을 거쳐야 하므로 **안전성**이 높아집니다.

## 🎯 주요 사용 시나리오 (When to Use)

| 시나리오 | 설명 | 구체적 예시 |
|---------|------|-----------|
| 시나리오 1 | **특정 포트 접근 제한**이 필요한 경우 | 기업이 외부에서 SSH(22번 포트) 접근을 제한하고, 내부 서버만 허용하는 경우 |
| 시나리오 2 | **다중 서브넷 간 트래픽 제어**가 필요한 경우 | 웹 서버와 데이터베이스 서버가 다른 서브넷에 위치할 때, 트래픽을 제한하는 경우 |
| 시나리오 3 | **특정 IP 범위의 트래픽 허용**이 필요한 경우 | 파트너사 IP 범위만 허용하고, 나머지는 거부하는 경우 |

**이럴 때 NACL를 선택하세요:**
- ✅ **상황 1:** 보안 그룹이 충분하지 않거나, 더 세밀한 네트워크 제어가 필요할 때
- ✅ **상황 2:** 서브넷 간 트래픽을 제어해야 할 때
- ✅ **상황 3:** 특정 IP 범위의 트래픽을 명시적으로 허용해야 할 때

**이럴 때는 다른 서비스를 고려하세요:**
- ❌ **상황 → 보안 그룹 사용:** 상태 기반 규칙이 필요할 때, 보안 그룹이 더 적합합니다.
- ❌ **상황 → Network Firewall 사용:** 애플리케이션 레이어에서 트래픽을 필터링해야 할 때, Network Firewall이 더 적합합니다.
- ❌ **상황 → WAF 사용:** HTTP/HTTPS 트래픽에서 규칙을 적용해야 할 때, WAF가 더 적합합니다.

## 📚 핵심 개념

### 개념 1: **NACL의 기본 개념 및 역할**
NACL(Network Access Control List)는 VPC 내에서 네트워크 트래픽을 제어하는 **stateless** 방식의 보안 메커니즘입니다. NACL은 서브넷 수준에서 트래픽을 필터링하며, 보안 그룹과 병행하여 사용됩니다. 보안 그룹은 인스턴스 수준에서 트래픽을 제어하지만, NACL은 보안 그룹이 실패했을 때 **보조 보안 계층**으로 작용합니다. NACL은 **입/출 트래픽을 명시적으로 허용/차단**하며, 모든 트래픽이 NACL의 규칙에 부합해야 합니다.  
이 개념은 네트워크 보안을 **이중화**하여 위험을 최소화하는 데 핵심적인 역할을 합니다. 특히, 보안 그룹이 제대로 작동하지 않을 경우, NACL이 트래픽을 차단하는 데 기여합니다.

#### 왜 중요한가?
- **보안 격차 보완**: 보안 그룹이 실패했을 때 NACL이 마지막 방어선으로 작용합니다.  
- **서브넷 수준 제어**: 서브넷 전체 트래픽을 단일 규칙으로 관리할 수 있어 대규모 네트워크에서 효율적입니다.  
- **필수적인 보안 계층**: AWS에서 보안 그룹과 NACL을 병행하여 사용하는 것이 권장되는 이유입니다.

#### 세부 요소
| 요소 | 설명 | 예시 |
|-----|-----|-----|
| **Stateless 특성** | 트래픽이 허용/차단 되는 경우, 관련된 트래픽도 별도로 설정해야 합니다. | SSH 연결을 허용할 경우, 응답 트래픽도 명시적으로 허용해야 합니다. |
| **서브넷 수준 적용** | 특정 서브넷에만 NACL을 적용할 수 있습니다. | 개발 서브넷에만 NACL을 설정하여 외부 트래픽을 제어합니다. |
| **기본 차단 규칙** | 모든 트래픽이 기본적으로 차단되며, 명시적으로 허용해야 합니다. | 기본 규칙으로 모든 트래픽을 차단하고, SSH 포트만 허용합니다. |

> **💡 Tip:** NACL은 보안 그룹의 보조 수단으로 사용되며, 보안 그룹이 실패할 경우 NACL이 마지막 방어선이 됩니다. 따라서 중요한 서브넷에 대해 NACL을 필수적으로 설정해야 합니다.

---

### 개념 2: **VPC Flow Logs의 작동 원리**
VPC Flow Logs는 VPC 내 트래픽 흐름을 로깅하는 기능입니다. 이 로그는 IP 트래픽의 **원천, 목적지, 포트, 상태** 등을 기록하여 네트워크 문제 진단과 보안 모니터링에 활용됩니다. VPC Flow Logs는 AWS CloudWatch와 연동되어 실시간으로 로그를 분석할 수 있습니다.  
이 기능은 네트워크 트래픽의 **투명성과 가시성**을 높여, 보안 위협 탐지 및 성능 최적화에 중요한 역할을 합니다.

#### 작동 원리
1. **트래픽 모니터링**: VPC 내 모든 리소스(인스턴스, 로드 밸런서 등)의 입/출 트래픽을 감시합니다.  
2. **로그 생성**: 트래픽 정보를 JSON 형식의 로그로 변환하여 CloudWatch에 전송합니다.  
3. **분석 및 대응**: 로그를 통해 이상 트래픽을 식별하고, 보안 정책을 조정하거나 네트워크 문제를 해결합니다.

> **💡 Tip:** VPC Flow Logs는 보안 감사 및 네트워크 트래픽 분석에 필수적입니다. 특히, 의심스러운 트래픽이 발생할 경우 로그를 통해 원인을 파악할 수 있습니다.

### 개념 3: **Network Firewall의 주요 특징**
Network Firewall은 AWS에서 제공하는 **관리형 방화벽 서비스**로, VPC 내에서 트래픽을 필터링하는 데 사용됩니다. 이 서비스는 **stateful** 방식으로 작동하며, 보안 그룹과 NACL을 보완하는 보안 계층으로 사용됩니다. Network Firewall은 **신규 트래픽의 원천 IP, 포트, 프로토콜을 기반으로 규칙을 적용**합니다.  
이 기능은 고급 보안 요구사항을 충족시키는 데 중요한 역할을 하며, 특히 **공격 시도를 탐지하고 차단**하는 데 효과적입니다.

#### 주요 특징
1. **Stateful 방식**: 트래픽이 허용된 후에도 관련된 트래픽을 자동으로 허용합니다.  
2. **고유한 규칙 세트**: 특정 IP 범위, 포트, 프로토콜에 따라 트래픽을 필터링할 수 있습니다.  
3. **관리형 서비스**: AWS가 관리하며, 사용자는 규칙을 설정하고 모니터링할 수 있습니다.

> **💡 Tip:** Network Firewall은 보안 그룹과 NACL을 보완하는 데 사용되며, **고위험 트래픽을 차단**하는 데 효과적입니다. 특히, 악성 IP 또는 특정 포트의 트래픽을 차단해야 할 경우 필수적으로 활용할 수 있습니다.

## 🖥️ AWS 콘솔에서 NACL 사용하기

### Step 1: NACL 서비스 접속  
1. AWS Management Console에 로그인합니다  
   - URL: https://console.aws.amazon.com  
   - AWS 계정이 생성되어 있어야 합니다.  
2. 상단 검색창에 **"Network Access Control List"** 또는 **"NACL"**을 입력합니다.  
   - 검색 결과에서 **"Network Access Control Lists"**를 선택합니다.  
3. **"NACL"** 대시보드로 이동합니다.  

> **📸 화면 확인:** NACL 목록이 표시되고, **"Create NACL"** 버튼이 보일 때 정상입니다.  
> **💡 Tip:** NACL은 VPC 내 서브넷에 대한 트래픽 통제 역할을 합니다.  

---

### Step 2: [주요 작업 1 - 리소스 생성]  
1. **NACL 생성**  
   - **"Create NACL"** 버튼을 클릭합니다.  
   - **Subnets** 필드에서 생성할 서브넷을 선택합니다 (예: default VPC의 public subnet).  
   - **Tags**에 이름을 입력합니다 (예: `NACL-Web-Access`).  
   - **"Create"** 버튼을 클릭하여 생성합니다.  

2. **Inbound/Outbound 규칙 설정**  
   - 생성된 NACL을 선택합니다.  
   - **"Inbound Rules"** 탭에서 **"Edit"** 버튼을 클릭합니다.  
   - 규칙을 추가합니다:  
     - **Rule Number:** 100  
     - **Protocol:** TCP  
     - **Port Range:** 80  
     - **Source:** `0.0.0.0/0` (모든 IP)  
   - **"Outbound Rules"** 탭에서 동일한 방식으로 포트 443을 허용합니다.  
   - **"Save"** 버튼을 클릭합니다.  

3. **확인 사항**  
   - 생성된 NACL이 **"Associated Subnets"**에 정상적으로 연결되어 있는지 확인합니다.  
   - 규칙이 **"Active"** 상태인지 확인합니다.  

> **📸 화면 확인:** NACL 목록에서 생성된 NACL이 표시되고, **"Inbound/Outbound Rules"** 탭에서 규칙이 적용된 상태인지 확인합니다.  

---

### Step 3: [주요 작업 2 - 설정/구성]  
1. **규칙 순서 조정**  
   - NACL 규칙은 **순서대로** 적용됩니다.  
   - 예: 규칙 100이 먼저 적용되며, 이후 규칙은 100보다 큰 번호여야 합니다.  
   - 필요 시 **"Edit"** 버튼으로 규칙 순서를 조정합니다.  

2. **규칙 수정 및 삭제**  
   - **"Inbound/Outbound Rules"** 탭에서 특정 규칙을 선택 후 **"Edit"** 또는 **"Delete"**를 클릭합니다.  
   - 수정 시 **"Save"** 버튼을 클릭하여 변경 사항을 저장합니다.  

3. **NACL과 서브넷 연결**  
   - **"Subnets"** 탭에서 서브넷을 선택하고 **"Associate"** 버튼을 클릭합니다.  
   - 연결된 서브넷의 트래픽이 NACL 규칙에 따라 제어됩니다.  

> **⚠️ 주의:** NACL은 **stateless**입니다. 보안 그룹과 달리 응답 트래픽이 자동으로 허용되지 않으므로, 양방향 규칙을 설정해야 합니다.  

---

### Step 4: 설정 확인 및 테스트  
1. **생성된 리소스 확인 방법**  
   - **"NACLs"** 목록에서 생성한 NACL을 선택합니다.  
   - **"Description"** 탭에서 서브넷 연결 상태와 규칙 정보를 확인합니다.  

2. **상태 확인 방법**  
   - **"Status"** 섹션에서 **"Active"** 상태인지 확인합니다.  
   - **"Actions"** 메뉴에서 **"Delete"**를 클릭해 리소스 상태를 변경할 수 있습니다.  

3. **정상 동작 테스트 방법**  
   - **VPC Flow Logs**를 사용해 트래픽 흐름을 확인합니다:  
     ```bash
     aws ec2 describe-flow-logs --flow-log-ids <flow-log-id>
     ```  
   - 테스트 인스턴스에서 웹 브라우저로 외부 서버에 접속해 포트 80/443 접근이 제한되는지 확인합니다.  
   - **"Test"** 버튼을 클릭해 특정 IP/포트의 접근이 허용/거부되는지 테스트합니다.  

> **💡 Tip:** NACL는 기본적으로 모든 트래픽을 거부합니다. 규칙을 명확히 설정하고, 정기적으로 검토해야 합니다.  
> **⚠️ 주의:** NACL은 **비용이 발생하지 않지만**, 규칙 오류로 인해 보안 취약점이 발생할 수 있으므로 주의해야 합니다.

### 예제 1: NACL 리소스 조회
```bash
# [NACL 목록 조회]
aws ec2 describe-network-acls --query 'NetworkAcls[].{ID:NetworkAclId,Subnets:Associations[].SubnetId}' --output table
```

**옵션 설명:**
| 옵션 | 설명 | 예시 값 |
|-----|------|--------|
| `--query` | 필터링 표현식 (NetworkAclId와 SubnetId 추출) | `'NetworkAcls[].{ID:NetworkAclId,Subnets:Associations[].SubnetId}'` |
| `--output` | 출력 형식 | `table` (테이블 형식으로 보기) |

**예상 출력:**
```
| ID        | Subnets             |
|-----------|---------------------|
| acl-12345 | subnet-67890        |
| acl-54321 | subnet-09876        |
```

---

### 예제 2: NACL 리소스 생성
```bash
# [NACL 생성 (서브넷 연결)]
aws ec2 create-network-acl --subnet-ids subnet-12345 --tag-specifications "ResourceType=network-acl,Tags=[{Key=Name,Value=example-nacl}]"
```

**필수 옵션:**
- `--subnet-ids`: NACL을 연결할 서브넷 ID (복수 가능)
- `--tag-specifications`: 리소스 이름 태깅 (관리 용도)

**예상 출력:**
```json
{
    "NetworkAcl": {
        "NetworkAclId": "acl-54321",
        "SubnetAssociations": [
            {
                "SubnetId": "subnet-12345",
                "RouteTableAssociationId": ""
            }
        ]
    }
}
```

---

### 예제 3: NACL 리소스 수정
```bash
# [NACL 규칙 추가]
aws ec2 create-network-acl-rule \
    --network-acl-id acl-12345 \
    --rule-number 100 \
    --action allow \
    --protocol tcp \
    --rule-action allow \
    --port-range FromPort=80,ToPort=80
```

**수정 가능한 항목:**
- 규칙 번호 (`--rule-number`)
- 허용/거부 (`--rule-action`)
- 프로토콜 및 포트 범위 (`--protocol`, `--port-range`)
- IP 범위 (`--cidr-block`)

### 예제 4: NACL 리소스 삭제
```bash
# [NACL 삭제]
aws ec2 delete-network-acl --network-acl-id acl-12345

# 삭제 확인 (존재 여부 확인)

```

> **⚠️ 주의:** NACL 삭제는 되돌릴 수 없습니다. 먼저 관련 서브넷 연결을 해제해야 합니다.  
> `aws ec2 disassociate-subnet-association --association-id assoc-12345` 명령어로 연결 해제 가능.

### CLI 사용 팁
1. **리전 설정 필수:** `aws configure set region us-east-1` 명령어로 리전 변경  
2. **태그 관리:** `--tag-specifications` 옵션으로 NACL 이름 태깅  
3. **규칙 순서:** 규칙 번호가 낮을수록 우선순위가 높습니다.  
4. **삭제 전 확인:** `describe-network-acls` 명령어로 연결된 서브넷 확인  

> **💡 Tip:** NACL은 **stateless**이므로, 모든 규칙을 명시적으로 설정해야 합니다. 보안 그룹과 달리 대응 규칙이 자동 생성되지 않습니다.

## 🎯 SAA-C03 시험 핵심 포인트

### 시험에서 자주 출제되는 NACL 포인트

#### 📌 핵심 출제 포인트 TOP 5
1. **NACL vs 보안 그룹 차이점**  
   - 설명: NACL은 서브넷 수준에서 무상태 방화벽 역할을 하며, 보안 그룹은 인스턴스 수준에서 상태ful 방화벽 역할을 합니다. 시험에서 두 서비스의 차이점과 적절한 사용 시나리오를 구분하는 문제가 자주 출제됩니다.  
   - 키워드: `무상태 방화벽`, `서브넷 수준`, `상태ful 방화벽`, `인스턴스 수준`

2. **NACL의 기본 규칙 (Default Deny)**  
   - 설명: NACL은 기본적으로 모든 트래픽을 거부하는 정책을 따르며, 명시적으로 허용된 트래픽만 통과합니다. 이 개념은 보안 정책의 기본 원칙으로 시험에서 필수적으로 이해해야 합니다.  
   - 키워드: `기본 거부`, `명시적 허용`, `보안 정책`, `default deny`

3. **규칙 순서의 중요성**  
   - 설명: NACL의 규칙은 상위에서 하위로 순차적으로 검사되며, 동일한 트래픽을 허용/거부하는 규칙이 존재할 경우 상위 규칙이 우선 적용됩니다. 규칙 순서 오류는 실무 및 시험에서 큰 실수로 이어질 수 있습니다.  
   - 키워드: `규칙 순서`, `상위 규칙`, `상태less`, `트래픽 우선순위`

4. **NACL의 상태less 특성**  
   - 설명: NACL은 상태less 방화벽으로, 각 패킷이 독립적으로 검사되며, 보안 그룹과 달리 응답 패킷을 자동으로 허용하지 않습니다. 이 점은 네트워크 보안 설계에 핵심적인 차이로 시험에 자주 출제됩니다.  
   - 키워드: `상태less`, `패킷별 검사`, `응답 패킷`, `상태ful vs 상태less`

5. **NACL의 제한사항 및 한계**  
   - 설명: NACL은 서브넷 수준에서만 작동하며, 인스턴스 수준 보안은 보안 그룹으로 관리해야 합니다. 또한, IP 주소 범위를 정확히 설정해야 하며, 실수로 범위를 넓게 설정하면 보안 위험이 발생할 수 있습니다.  
   - 키워드: `서브넷 수준`, `IP 범위`, `보안 그룹`, `정확한 설정`

#### ⚠️ 시험에서 자주 나오는 함정
| 함정 유형 | 설명 | 올바른 답변 |
|----------|------|------------|
| 함정 1 | "NACL은 상태ful 방화벽이므로 응답 패킷을 자동으로 허용한다"는 문장이 출제될 수 있음 | ❌ 오답. NACL은 상태less 방화벽으로, 응답 패킷을 자동으로 허용하지 않음 |
| 함정 2 | "NACL은 보안 그룹보다 더 많은 규칙을 지원한다"는 문장이 출제될 수 있음 | ❌ 오답. NACL은 보안 그룹보다 규칙 수 제한이 더 많음 |
| 함정 3 | "NACL은 기본적으로 모든 트래픽을 허용한다"는 문장이 출제될 수 있음 | ❌ 오답. NACL은 기본적으로 모든 트래픽을 거부하는 'default deny' 정책을 따름 |

#### 🔄 NACL vs 비슷한 서비스 비교 (시험 단골!)
| 비교 항목 | NACL | 대안 서비스 | 선택 기준 |
|----------|----------|-----------|----------|
| **용도** | 서브넷 수준에서 무상태 방화벽으로 트래픽 제어 | Security Groups (인스턴스 수준 상태ful 방화벽) | 인스턴스별 보안이 필요한 경우 |
| **확장성** | 규칙 수 제한이 많음 (50개까지) | Security Groups (더 많은 규칙 지원) | 고유한 규칙이 필요한 경우 |
| **비용** | 보안 그룹과 동일한 비용 구조 | VPC Flow Logs (트래픽 로깅 추가 비용 발생) | 로깅이 필요한 경우 |
| **지연시간** | 상태less로 인해 처리 지연이 발생 | Network Firewall (고성능 방화벽) | 실시간 보안 검사가 필요한 경우 |
